# Agent-Mem — Snapshot Quality Analysis

*A problem statement document. Captures what we observed about current claude-mem output quality and what we want instead. This is the "why" behind Phase 1 decisions.*

---

## Terminology Note

Claude-Mem calls these "session summaries". In this project they are **snapshots** — point-in-time captures generated after each Claude response, not at session end. In the viewer UI they are **cards**. The word "summary" is a misnomer — these aren't summaries of a full session, they're captures of a single response cycle. See `01-project-summary.md`.

---

## What We Observed

### The Viewer at localhost:37777

The current viewer mixes three card types in one chronological feed:
- **PROMPT cards** — the user's raw messages to Claude, shown in full including injected XML tags like `<ide_opened_file>`, `<task-notification>` etc.
- **DISCOVERY cards** — observations generated when Claude reads files or explores the codebase
- **SNAPSHOT cards** (upstream calls these SESSION SUMMARY cards) — the structured captures generated after each response

By default all three are visible, making the feed very noisy. The most common view is a wall of PROMPT cards with the occasional useful card buried in between.

---

## Problem 1 — PROMPT Cards Are Noise

PROMPT cards show the raw user messages including system-injected XML. Example:
```
<ide_opened_file>The user opened the file Untitled-1 in the IDE...</ide_opened_file>
I want to check that the tax calculator we're currently using is 100% correct.
```

**Value to a future agent:** None. The agent doesn't need to know what the user typed — it needs to know what was decided and built.

**Fix:** Hide PROMPT cards by default. Add a toggle to show them when needed (Phase 2 UI).

---

## Problem 2 — DISCOVERY Cards Are Low Signal

Discovery observations are generated by the PostToolUse hook whenever Claude reads a file or explores the codebase. Example:

> **Locating Tax Strategy Project Reports**
> - The user is looking for an HTML output report.
> - The report is associated with the project located at docs/projects/tax-strategy-fy23-25.

**Value to a future agent:** Almost none. This just says "Claude looked at a directory." It doesn't tell me anything I couldn't infer from context.

**Additional problem:** The narrative tab is vague even when it has more text:
> "The user has opened a specific markdown file related to expense updates within the tax strategy project..."

It should say the actual file path but instead gives a vague description. The file path data IS available in the tool call — the memory agent just didn't use it.

**Fix (Phase 1):** Remove `discovery` as an observation type from the custom mode entirely. If exploration reveals something genuinely non-obvious (a gotcha, an architectural pattern), it should be logged as a `decision` or `change` instead.

**Fix (Phase 2):** If we keep discovery cards for any reason, they should be hidden by default and shown only via filter.

---

## Problem 3 — Snapshot Fields Are Generic

Current snapshot fields: `request`, `investigated`, `learned`, `completed`, `next_steps`, `notes`.

### What's good:
The snapshot card format is actually the most useful thing in the viewer. When populated well, it gives a future agent a real picture of what happened. Example of a good snapshot (Session #29):

> **Decide on project structure for tax strategy analysis**
> - Investigated: Placement of artifacts debated — `projects/` vs `docs/projects/`
> - Learned: Analytical work belongs under `docs/` not `src/`
> - Completed: Decision made to place artifacts in `docs/projects/tax-strategy-fy23-25/`
> - Next Steps: Create `brief.md`, `scenarios.md`, and `reports/` subdirectory

A future agent reading this knows exactly what was decided and what to do next. This is genuinely useful.

### What's not good:

**`investigated` is usually filler.** Example (Session #25):
> "The file structure of the 'hd-xero-connect' repository was explored by listing the contents of the 'docs' and 'src' directories."

This says nothing useful. It just logged that directories were listed.

**There's no decision capture field.** Decisions get buried in `completed` or `learned` with no structure. A future agent has to parse prose to find them.

**There's no mistakes field.** If something went wrong and had to be reversed, there's no dedicated place to capture it.

**There's no trade-off field.** "We chose X over Y because Z" is the most useful thing for a future agent to know, but it gets lost in filler text.

**`completed` duplicates git commit messages.** When commit discipline is good, the commit message is ground truth. A future agent can `git show` it. Storing a prose version alongside is redundant.

**`next_steps` is ambiguous.** Agents can't tell "do this now" from "do this someday". Replaced by `open_questions` (pending decisions) and `unresolved` (dangling state).

---

## Problem 4 — Superseded Decisions Not Resolved

The most critical problem: when multiple decisions are made in the same session or across sessions, and one supersedes another, all versions appear in the record.

**Real example observed:**

Session #28 decided: place tax strategy project under `projects/` at root level (Option A)
Session #29 decided: place tax strategy project under `docs/projects/` (reversed Session #28)

Both snapshot cards appear in the viewer. A future agent reading both gets contradictory instructions. Unless it reads carefully and notices the date order, it might follow the older decision.

**What we want:** The snapshot prompt must explicitly instruct the memory agent: "If a decision in this snapshot reverses or supersedes an earlier decision, note that explicitly. Only record the final decision as active."

---

## Problem 5 — "Session Summary" Is a Misnomer

The Stop hook fires after every Claude response — not at session end. A long multi-turn session can produce dozens of snapshot cards. Calling them "session summaries" implies a single end-of-session retrospective, which is not what they are.

**They are snapshots:** point-in-time captures after each response cycle. For short chats (the intended workflow), the last snapshot IS effectively the session record. For long sessions, there are many.

**Implication:** The snapshot prompt says "this is a checkpoint, the session is ongoing." This is correct behaviour. The custom mode prompt should be written with the assumption that most snapshots ARE final captures (due to the short-chat philosophy), while still being technically correct for multi-turn sessions.

---

## What Good Snapshot Output Looks Like

Based on our analysis, here's what a high-value snapshot card should look like for a future agent:

```
TITLE: Chose docs/projects/ directory structure for tax strategy work

DECISION LOG:
- Chose docs/projects/tax-strategy-fy23-25/ over projects/tax-strategy-fy23-25/ at root
  (Reversed decision from previous session — analytical/documentation work belongs in docs/, not a top-level projects/ dir)

DECISION TRADE-OFFS:
- Option A (projects/ at root): Better for bounded workstreams with code generation
- Option B (docs/projects/): Cleaner separation — docs are docs, src is code
- Chose B because this workstream produces reports and documentation, not application code

CONSTRAINTS:
- Analytical/documentation workstreams go under docs/, not projects/ at root

MISTAKES:
- None this snapshot

GOTCHAS:
- None discovered

COMMIT:
- feat: scaffold docs/projects/tax-strategy-fy23-25 directory structure

OPEN QUESTIONS:
- Verify whether HTML reports should be version-tracked or gitignored

UNRESOLVED:
- Contractor-level expense view task — not yet confirmed in scope
```

Compare this to the current output. The above tells a future agent exactly what to know, what was rejected, and why. No filler. No vague exploration logs.

## Summary of Fixes Required

| Problem | Phase | Fix |
|---|---|---|
| PROMPT cards visible by default | Phase 2 | Hide by default, add toggle |
| DISCOVERY cards are noise | Phase 1 | Remove discovery type from custom mode |
| `investigated` field is filler | Phase 1 | Drop from prompt |
| No decision capture field | Phase 1 | Add `decision_log` field |
| No trade-off capture | Phase 1 | Add `decision_trade_offs` field |
| No mistake capture | Phase 1 | Add `mistakes` field + observation type |
| No constraints field | Phase 1 | Add `constraints_log` field |
| No gotchas field | Phase 1 | Add `gotchas` field |
| `completed` duplicates commits | Phase 1 | Replace with `commit_ref` — link to commit hash/message |
| `next_steps` is ambiguous | Phase 1 | Split into `open_questions` and `unresolved` |
| Superseded decisions not resolved | Phase 1 | Update snapshot prompt explicitly |
| File paths vague in narratives | Phase 1 | Update observation prompt to require exact paths |
| No way to hide irrelevant fields | Phase 3 | Checkbox curation with `hidden_fields` column |
| "Session summary" is a misnomer | Ongoing | Use "snapshot" / "card" terminology throughout |
